# 先验证再优化

> 来源：stock-monitor 开发教训

## 核心原则

**在拿到闭环验证结果之前，持续优化是盲目改——你不知道改对了还是改错了。**

## 反模式

大量 commit 全在铺基础设施、调参数、加功能，但没见过一次完整的闭环验证。所有 outcome 都是 pending，核心指标没有基线。

## 正确做法

1. 确定核心指标（如 AI 决策加权准确率）
2. 跑一段时间（如 1 个月）积累数据
3. 拿到基线数据后再决定下一步
   - 指标达标 → 系统有价值，继续迭代
   - 指标不达标 → 检查根本原因（规则？prompt？数据？）

## 计算纪律

**任何涉及数字的计算——永远不要心算、不要靠记忆。** 使用 `python3 -c` 或 `node -e` 验证：

```bash
python3 -c "r=0.0356; n=252/80; print(f'{(1+r)**n-1:.1%}')"
python3 -c "import statistics; ..."
```

哪怕看着再简单的四则运算，也代码验证。
