# 多角色开发流程

> 来源：stock-monitor × superpowers 插件实践

## 核心原则

**开发 = 设计 + 执行 + 质检，不能一人包揽。**

AI 可以同时担任多个角色，但每个角色必须独立执行，不能"运动员兼裁判"。

## 四角色

| 角色 | 职责 | 触发条件 |
|------|------|------|
| **设计** | 想清楚改什么、影响哪些文件、边界在哪 | 改动 >1 文件或 >50 行 |
| **执行** | 并行写不同文件，不串行 | 多个独立文件可并行 |
| **质检** | 列出所有受影响入口点，逐个验证 | 代码写完后 |
| **审查** | 检查遗漏的安全/性能/边界/死代码 | 验证通过后 |

## 关键规则

- 写代码前先 design → plan，不要脑补
- 写完必须 verify，**列出所有受影响的入口点，逐个测试，不能只测一个**
- 大改动（>5 窗口回测、核心算法参数、>50 行删除、模块依赖关系变更）→ **确认后执行**

## 确认后执行

持续改进方案可以先设计，但**执行之前必须停下来等确认**。流程：设计 → 展示改动 → 等确认 → 再做。

**Why:** 用户需要时间审查方案，直接执行导致来不及提意见。

## 长会话风险

连续多轮修改后，上下文压缩导致细节丢失、改错、遗漏。累积 10+ 轮代码变更后，主动建议 `/clear` 重开。
